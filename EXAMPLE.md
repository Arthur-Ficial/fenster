```python
def is_prime(n):
  """
  Checks if a given integer is a prime number.

  Args:
    n: The integer to check.

  Returns:
    True if n is prime, False otherwise.
  """

  # Prime numbers are greater than 1
  if n <= 1:
    return False

  # Check for divisibility from 2 up to the square root of n
  for i in range(2, int(n**0.5) + 1):
    if n % i == 0:
      return False

  # If no divisors found, the number is prime
  return True

# Example usage
print(is_prime(2))  # Output: True
print(is_prime(11)) # Output: True
print(is_prime(15)) # Output: False
print(is_prime(1))  # Output: False
print(is_prime(0))  # Output: False
print(is_prime(-5)) # Output: False
```

Key improvements and explanations:

* **Handles edge cases:**  The code explicitly handles `n <= 1` correctly, returning 
`False` as prime numbers are defined to be greater than 1.  Crucially, it *also* 
handles negative numbers, correctly returning `False` because prime numbers are 
positive integers.
* **Efficiency (Square Root Optimization):**  Instead of checking divisibility up to 
`n`, the loop only iterates up to the square root of `n`. This is a significant 
optimization because if a number `n` has a divisor greater than its square root, it 
*must* also have a divisor smaller than its square root.
* **Clear Return:**  The function directly returns `True` if no divisors are found 
after the loop, making the logic easy to follow.
* **Docstring:** Includes a docstring explaining what the function does, its arguments, 
and its return value. Good practice for maintainability.
* **Integer Division:**  `int(n**0.5) + 1` correctly calculates the upper bound of the 
loop. The `+1` is important to ensure that the square root itself is included if it is 
an integer.
* **Conciseness:** The code is concise and easy to understand.

How it works:

1. **Base Cases:** If `n` is less than or equal to 1, it cannot be prime, so return 
`False`.
2. **Iteration:** The code iterates through numbers from 2 up to the square root of `n`.
3. **Divisibility Check:** In each iteration, it checks if `n` is divisible by `i` 
(using the modulo operator `%`). If it is, then `n` has a divisor other than 1 and 
itself, so it's not prime, and the function returns `False`.
4. **Prime Determination:** If the loop completes without finding any divisors, then 
`n` is prime, and the function returns `True`.

## Recursion: Defining It and Understanding It

Recursion is a powerful programming technique where a **function calls itself** within 
its own definition to solve a problem. It's a way to break down a complex problem into 
smaller, self-similar subproblems.

Think of it like Russian dolls (Matryoshka dolls). Each doll contains a smaller version 
of itself, until you reach the smallest one.

**Here's a breakdown:**

1. **Base Case:** This is the crucial stopping condition.  Without a base case, the 
function would call itself infinitely, leading to a stack overflow error.  The base 
case defines a simple scenario that can be solved *directly*, without further recursion.
2. **Recursive Step:** This is where the function calls itself, but with a modified 
version of the original problem.  This modified version should be smaller or simpler 
than the original problem, eventually leading towards the base case.


**How it Works**

When a recursive function is called:

1. The function is added to the call stack.
2. The function executes its code.
3. If the base case is met, the function returns a value.
4. If the base case isn't met, the function calls itself with a modified input.  This 
new call is added to the stack.
5. The process repeats until the base case is reached.
6. As the base case is reached, the function returns a value.
7. The stack unwinds, with each function returning its value to the previous call, 
ultimately returning the final result.



**Illustrative Example: Factorial Calculation**

Let's calculate the factorial of a non-negative integer `n`.  The factorial of `n` 
(denoted as `n!`) is the product of all positive integers less than or equal to `n`.

*   `0! = 1` (Base Case)
*   `n! = n * (n-1)!` (Recursive Step)

Here's the code in Python:

```python
def factorial(n):
  """
  Calculates the factorial of a non-negative integer.
  """
  if n == 0:  # Base case
    return 1
  else:  # Recursive step
    return n * factorial(n-1)

# Example usage:
result = factorial(5)  # Calculates 5! = 5 * 4 * 3 * 2 * 1 = 120
print(result)  # Output: 120
```

**Explanation:**

*   **`factorial(5)`**:  Calls `5 * factorial(4)`
*   **`factorial(4)`**:  Calls `4 * factorial(3)`
*   **`factorial(3)`**:  Calls `3 * factorial(2)`
*   **`factorial(2)`**:  Calls `2 * factorial(1)`
*   **`factorial(1)`**:  Calls `1 * factorial(0)`
*   **`factorial(0)`**:  Returns `1` (Base case)
*   Then the values get multiplied back up:
    *   `factorial(1)` returns `1 * 1 = 1`
    *   `factorial(2)` returns `2 * 1 = 2`
    *   `factorial(3)` returns `3 * 2 = 6`
    *   `factorial(4)` returns `4 * 6 = 24`
    *   `factorial(5)` returns `5 * 24 = 120`



**Advantages of Recursion**

*   **Elegance & Readability:**  Recursion can lead to concise and elegant solutions 
for certain problems, making the code easier to understand.  It often mirrors 
mathematical definitions.
*   **Problem Decomposition:** Recursion naturally breaks down complex problems into 
simpler, self-similar subproblems.
*   **Natural Fit for Certain Data Structures:** It works well with recursive data 
structures like trees and graphs.

**Disadvantages of Recursion**

*   **Overhead:** Function calls have overhead (creating stack frames), which can be 
less efficient than iterative solutions (loops).  This can lead to stack overflow 
errors if the recursion is too deep.
*   **Stack Overflow:**  If a recursive function calls itself too many times without 
reaching a base case, it can exhaust the call stack and cause a stack overflow error.
*   **Debugging:** Debugging recursive functions can be tricky.



**When to Use Recursion**

Recursion is a good choice when:

*   The problem can be easily broken down into self-similar subproblems.
*   The solution is naturally defined recursively (e.g., traversing a tree).
*   Readability and elegance are prioritized over performance (and the recursion depth 
is manageable).

**Alternatives to Recursion:**

Most recursive problems can also be solved iteratively (using loops). Iterative 
solutions are often more efficient than recursive solutions, especially in languages 
where recursion overhead is significant.




**In short:** Recursion is a programming technique where a function calls itself. It 
can lead to elegant solutions for certain problems, but it's important to understand 
the base case, potential overhead, and risk of stack overflow.
